# 理解 Ralph 文件

运行 `ralph-enable`、`ralph-import` 或 `ralph-setup` 后，您将拥有一个包含多个文件的 `.ralph/` 目录。本指南解释每个文件的作用，以及您是否需要编辑它。

## 文件参考表

| 文件 | 自动生成？ | 谁编写 | 谁读取 | 您应该... |
|------|-----------------|---------------|--------------|---------------|
| `.ralph/PROMPT.md` | 是 (带智能默认值) | **您** 自定义 | Ralph 每次循环读取 | 审查和自定义项目目标 |
| `.ralph/fix_plan.md` | 是 (可以导入任务) | **您** + Ralph 更新 | Ralph 读取和更新 | 添加/修改特定任务 |
| `.ralph/AGENT.md` | 是 (检测构建命令) | Ralph 维护 | Ralph 读取用于构建/测试 | 很少编辑 (自动维护) |
| `.ralph/specs/` | 创建空目录 | **您** 在需要时添加文件 | Ralph 读取获取上下文 | 当 PROMPT.md 不够详细时添加 |
| `.ralph/specs/stdlib/` | 创建空目录 | **您** 添加可重用模式 | Ralph 读取获取约定 | 添加共享模式和约定 |
| `.ralphrc` | 是 (项目感知) | 通常保持原样 | Ralph 启动时读取 | 很少编辑 (合理的默认值) |
| `.ralph/logs/` | 自动创建 | Ralph 写入日志 | 您审查用于调试 | 不要编辑 (只读) |
| `.ralph/status.json` | 运行时创建 | Ralph 更新 | 监控工具 | 不要编辑 (只读) |

## 核心文件

### PROMPT.md - 您的项目愿景

**目的**: Ralph 在每个循环开始时读取的高级指令。

**应包含的内容**:
- 项目描述和目标
- 关键原则或约束
- 技术栈和框架
- 质量标准

**不应包含的内容**:
- 分步实施任务 (使用 fix_plan.md)
- 详细 API 规范 (使用 specs/)
- 构建命令 (使用 AGENT.md)

**示例**:
```markdown
## 上下文
您是 Ralph，正在为书店库存系统构建 REST API。

## 关键原则
- 使用带异步数据库操作的 FastAPI
- 严格遵循 REST 约定
- 每个端点都需要测试
- 使用 OpenAPI 记录所有 API 端点
```

### fix_plan.md - 您的任务列表

**目的**: Ralph 处理的优先级任务清单。

**关键特征**:
- Ralph 完成后会勾选 `[x]` 项目
- Ralph 可能会添加它发现的新任务
- 您可以随时添加、重新排序或删除任务
- 更具体的任务 = 更好的结果

**好的任务结构**:
```markdown
## 优先级 1: 基础
- [ ] 为 Book 和 Author 创建数据库模型
- [ ] 使用异步支持设置 SQLAlchemy
- [ ] 为初始架构创建 Alembic 迁移

## 优先级 2: API 端点
- [ ] POST /books - 创建新书
- [ ] GET /books - 列出所有书籍并分页
- [ ] GET /books/{id} - 获取单本书及作者详情
```

**坏的任务结构**:
```markdown
- [ ] 让 API 工作
- [ ] 添加功能
- [ ] 修复错误
```

### specs/ - 详细规范

**目的**: 当 PROMPT.md 对某个功能不够详细时使用。

**何时使用 specs/**:
- 需要详细要求的复杂功能
- 必须严格遵循的 API 契约
- 具有特定验证规则的数据模型
- 外部系统集成

**何时不使用 specs/**:
- 简单的 CRUD 操作
- 已在 PROMPT.md 中解释清楚的功能
- 一般编码标准 (放在 PROMPT.md 中)

**示例结构**:
```
.ralph/specs/
├── api-contracts.md      # OpenAPI 风格的端点定义
├── data-models.md        # 实体关系和验证
└── third-party-auth.md   # OAuth 集成要求
```

### specs/stdlib/ - 标准库模式

**目的**: 项目的可重用模式和约定。

**属于这里的内容**:
- 错误处理模式
- 日志约定
- 通用实用函数规范
- 测试模式
- 代码风格决策

**示例**:
```markdown
# 错误处理标准

所有 API 错误必须返回：
{
  "error": {
    "code": "BOOK_NOT_FOUND",
    "message": "No book with ID 123 exists",
    "details": {}
  }
}

使用这些代码的 HTTPException：
- 400: 验证错误
- 404: 资源未找到
- 409: 冲突 (重复)
- 500: 内部错误 (记录完整跟踪)
```

### AGENT.md - 构建指令

**目的**: 如何构建、测试和运行项目。

**谁维护它**: 主要是 Ralph，因为它发现构建命令。

**您可能编辑的时机**:
- 为复杂项目设置初始构建命令
- 添加环境设置步骤
- 记录部署命令

### .ralphrc - 项目配置

**目的**: 特定于项目的 Ralph 设置。

**默认内容** (通常保持原样即可):
```bash
PROJECT_NAME="my-project"
PROJECT_TYPE="typescript"
MAX_CALLS_PER_HOUR=100
ALLOWED_TOOLS="Write,Read,Edit,Bash(git *),Bash(npm *),Bash(pytest)"
```

**编辑时机**:
- 限制工具权限以提高安全性
- 调整速率限制
- 更改会话超时

## 常见场景

### 场景 1: 简单功能添加

只需编辑 fix_plan.md：
```markdown
- [ ] 添加一个返回 {"status": "ok"} 的 /health 端点
```

### 场景 2: 具有特定要求的复杂功能

首先添加规范文件，然后添加任务：

1. 创建 `.ralph/specs/search-feature.md`：
```markdown
# 搜索功能规范

## 要求
- 对书名和描述进行全文搜索
- 必须支持：
  - 精确短语匹配: "lord of the rings"
  - 布尔运算符: fantasy AND epic
  - 模糊匹配容错
```

2. 然后添加到 fix_plan.md：
```markdown
- [ ] 根据 specs/search-feature.md 实现搜索
```

### 场景 3: 建立团队约定

添加到 specs/stdlib/：
```markdown
# 日志约定

所有服务方法必须记录：
- 带有参数的入口 (DEBUG 级别)
- 带有结果摘要的出口 (DEBUG 级别)
- 带有完整上下文的错误 (ERROR 级别)
```

## 成功提示

1. **简单开始** - 首先只使用 PROMPT.md 和 fix_plan.md。仅在需要时添加 specs/。

2. **具体化** - 模糊的要求产生模糊的结果。"添加用户 auth" 不如 "添加带有 /login 和 /logout 端点的 JWT 身份验证" 好。

3. **让 fix_plan.md 演变** - Ralph 会添加它发现的任务。定期审查并重新优先级。

4. **不要过度指定** - 如果 Claude 可以从上下文弄清楚，就不需要指定它。

5. **查看日志** - 当出现问题时，`.ralph/logs/` 告诉您 Ralph 在想什么。
