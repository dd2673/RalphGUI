# 编写有效需求

Ralph 在理解您想要什么时效果最好。本指南向您展示如何在 PROMPT.md 中编写清晰的需求。

## PROMPT.md: 好与坏示例

### 坏示例

```markdown
# 项目

做一个好的 API。使用最佳实践。
应该快速且运行良好。
```

**问题**:
- "东西"是什么？太模糊
- "最佳实践"是什么？Claude 会猜测
- "快速"和"运行良好"不可衡量

### 好示例

```markdown
# Ralph 开发指令

## 上下文
您是 Ralph，正在为宠物收容所构建 REST API。
该 API 管理动物、被收养者和收养记录。

## 技术栈
- Python 3.11+ 使用 FastAPI
- PostgreSQL 使用 SQLAlchemy
- pytest 用于测试

## 关键原则
- 遵循 RESTful 端点约定
- 所有端点需要身份验证（除 GET 外）
- 所有实体使用软删除
- 所有列表端点分页 (默认 20，最大 100)

## 数据实体
- Animal: name, species, breed, age, status
- Adopter: name, email, phone, approved
- Adoption: animal_id, adopter_id, date, status
```

**为什么这样有效**:
- 清晰的领域 (宠物收容所)
- 特定的技术选择
- 可衡量的约束 (分页限制)
- 具体的数据模型

## fix_plan.md: 任务编写

### 金发姑娘原则

任务应该**不太大，不太不小**。

**太大** (Ralph 不知道从哪里开始):
```markdown
- [ ] 构建整个身份验证系统
```

**太小** (浪费循环迭代):
```markdown
- [ ] 创建 auth 文件夹
- [ ] 创建 auth/__init__.py 文件
```

**刚刚好** (有意义的工作一个循环):
```markdown
- [ ] 创建 POST /login 和 POST /logout 端点
- [ ] 添加 JWT 令牌中间件
```

### 任务结构模板

```markdown
# 修复计划 - [项目名称]

## 优先级 1: 基础/关键路径
- [ ] [具体、可操作的任务]

## 优先级 2: 核心功能
- [ ] [功能任务]

## 优先级 3: 完善
- [ ] [增强]
```

## 常见错误

### 错误 1: 假设 Claude 知道您的偏好

**坏**:
```markdown
使用标准身份验证。
```

**好**:
```markdown
使用 JWT 身份验证，令牌在 1 小时后过期。
```

### 错误 2: 模糊的任务

**坏**:
```markdown
- [ ] 改进代码
```

**好**:
```markdown
- [ ] 为缺失的错误处理添加 try/catch
```
