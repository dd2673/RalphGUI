# 编写有效需求

Ralph 在理解您想要什么时效果最好。本指南向您展示如何在 PROMPT.md 中编写清晰的需求，何时使用 specs/，以及 fix_plan.md 在开发过程中如何演变。

## PROMPT.md: 好与坏示例

### 坏示例

```markdown
# 项目

为管理东西做一个好的 API。使用最佳实践。
应该快速且运行良好。
```

**问题**:
- "东西"是什么？太模糊。
- "最佳实践"是什么？Claude 会猜测。
- "快速"和"运行良好"不可衡量。

### 好示例

```markdown
# Ralph 开发指令

## 上下文
您是 Ralph，正在为宠物收容所构建 REST API。
该 API 管理动物、被收养者和收养记录。

## 技术栈
- Python 3.11+ 使用 FastAPI
- PostgreSQL 使用 SQLAlchemy (async)
- pytest 用于测试
- Pydantic 用于验证

## 关键原则
- 遵循标准约定的 RESTful 端点
- 除 GET /animals 外所有端点都需要身份验证
- 所有实体使用软删除 (is_deleted 标志，不是实际删除)
- 所有列表端点分页 (默认 20，最大 100)

## 数据实体
- Animal: name, species, breed, age, status (available/adopted/pending)
- Adopter: name, email, phone, approved (boolean)
- Adoption: animal_id, adopter_id, date, status

## 质量标准
- 每个端点至少需要一个 happy-path 测试
- 带有清晰错误消息的输入验证
- 所有端点的 OpenAPI 文档
```

**为什么这样有效**:
- 清晰的领域 (宠物收容所)
- 特定的技术选择
- 可衡量的约束 (分页限制)
- 具体的数据模型
- 定义的质量标准

## fix_plan.md: 任务编写

### 金发姑娘原则

任务应该**不太大，不太不小**。

**太大** (Ralph 不知道从哪里开始):
```markdown
- [ ] 构建整个身份验证系统
```

**太小** (浪费循环迭代):
```markdown
- [ ] 创建 auth 文件夹
- [ ] 创建 auth/__init__.py 文件
- [ ] 创建 auth/routes.py 文件
```

**刚刚好** (有意义的工作一个循环):
```markdown
- [ ] 创建带有 POST /login 和 POST /logout 端点的 auth 路由
- [ ] 添加 JWT 令牌生成和验证中间件
- [ ] 创建刷新令牌端点 POST /auth/refresh
```

### 任务结构模板

```markdown
# 修复计划 - [项目名称]

## 优先级 1: [基础/关键路径]
- [ ] [具体、可操作的任务]
- [ ] [另一个具体任务]

## 优先级 2: [核心功能]
- [ ] [功能任务]
- [ ] [功能任务]

## 优先级 3: [完善/锦上添花]
- [ ] [增强]
- [ ] [文档]

## 发现
<!-- Ralph 在这里添加它发现的任务 -->
```

### fix_plan.md 如何演变

**初始状态** (您编写):
```markdown
## 优先级 1: 数据库
- [ ] 为 Animal、Adopter、Adoption 设置数据库模型

## 优先级 2: API
- [ ] 为动物创建 CRUD 端点
```

**循环 1 后** (Ralph 更新):
```markdown
## 优先级 1: 数据库
- [x] 为 Animal、Adopter、Adoption 设置数据库模型

## 优先级 2: API
- [ ] 为动物创建 CRUD 端点

## 发现
- [ ] 使用 Alembic 添加数据库迁移
- [ ] 为测试数据库创建 pytest fixtures
```

**循环 3 后**:
```markdown
## 优先级 1: 数据库
- [x] 为 Animal、Adopter、Adoption 设置数据库模型

## 优先级 2: API
- [x] 为动物创建 CRUD 端点
- [ ] 为被收养者创建 CRUD 端点

## 发现
- [x] 使用 Alembic 添加数据库迁移
- [x] 为测试数据库创建 pytest fixtures
- [ ] 在 GET /animals 端点添加分页
```

Ralph 添加它发现的任务并在工作时勾选它们。您可以：
- 通过将任务移动到不同的优先级部分来重新排序任务
- 删除不再相关的任务
- 随时添加新任务

## 何时使用 specs/

### 为复杂功能使用 specs/

**PROMPT.md 说**:
```markdown
添加一个向被收养者建议动物的匹配算法。
```

**这太模糊了。** 创建 `.ralph/specs/matching-algorithm.md`:
```markdown
# 动物匹配算法

## 输入
- 被收养者偏好: species, max_age, size_preference
- 可用动物列表

## 算法
1. 按 species 过滤 (必需匹配)
2. 按年龄偏好评分 (0-100 分)
   - 在范围内: 100 分
   - 2 年内: 50 分
   - 之外: 0 分
3. 按大小偏好评分 (0-50 分)
4. 按总分返回前 5 名

## 输出格式
```json
[
  {"animal_id": 1, "score": 145, "reasons": ["species match", "age within preference"]},
  {"animal_id": 3, "score": 120, "reasons": ["species match"]}
]
```

## 边界情况
- 无匹配: 返回空数组
- 平分: 按 animal.created_at 排序 (最旧的优先)
```

**然后在 fix_plan.md 中**:
```markdown
- [ ] 根据 specs/matching-algorithm.md 实现匹配算法
```

### 为约定使用 specs/stdlib/

当您希望项目保持一致性时，记录它：

`.ralph/specs/stdlib/error-responses.md`:
```markdown
# 错误响应标准

所有 API 错误返回此结构：

```json
{
  "error": {
    "code": "ANIMAL_NOT_FOUND",
    "message": "No animal with ID 42 exists",
    "field": null,
    "details": {}
  }
}
```

## 错误代码
| 代码 | HTTP 状态 | 何时 |
|------|-------------|------|
| VALIDATION_ERROR | 400 | 无效输入 |
| NOT_FOUND | 404 | 资源不存在 |
| ALREADY_ADOPTED | 409 | 动物不可用 |
| UNAUTHORIZED | 401 | 缺失/无效令牌 |
```

### 不要为所有事情使用 specs/

**过度杀伤** - 您不需要 specs/ 用于:
```markdown
# 用户密码要求

密码必须至少 8 个字符。
```

**只需放在 PROMPT.md 中**:
```markdown
## 身份验证
- 密码: 最少 8 个字符，至少一个数字
- JWT 令牌在 1 小时后过期
```

## 常见错误

### 错误 1: 假设 Claude 知道您的偏好

**坏**:
```markdown
使用标准身份验证。
```

**好**:
```markdown
使用 JWT 身份验证，1 小时令牌过期。
刷新令牌持续 7 天并在使用时轮换。
```

### 错误 2: 混合实现与需求

**坏** (在 PROMPT.md 中):
```markdown
创建一个名为 auth.py 的文件并添加这些导入：
import jwt
from datetime import datetime
```

**好** (在 PROMPT.md 中):
```markdown
使用 JWT 进行身份验证。令牌应在 1 小时后过期。
```

让 Ralph 想出实现细节。

### 错误 3: 过度指定测试

**坏**:
```markdown
- [ ] 编写 test_create_animal_success
- [ ] 编写 test_create_animal_invalid_species
- [ ] 编写 test_create_animal_missing_name
- [ ] 编写 test_create_animal_negative_age
```

**好**:
```markdown
- [ ] 为动物创建编写测试 (成功和验证错误)
```

Ralph 知道如何编写测试。告诉它测试什么，而不是如何测试。

### 错误 4: 忘记"为什么"

**坏**:
```markdown
为所有 API 响应添加 100ms 延迟。
```

**好**:
```markdown
为所有 API 响应添加 100ms 延迟 (需要符合外部支付 API 的速率限制)。
```

当 Ralph 理解**为什么**时，它会做出更好的决定。

## 运行 Ralph 前检查清单

在 `ralph --monitor` 之前，验证：

- [ ] **PROMPT.md 有清晰的上下文** - Ralph 知道它在构建什么吗？
- [ ] **技术栈已指定** - 您选择了框架了吗？
- [ ] **关键约束已记录** - 身份验证方法？API 约定？
- [ ] **fix_plan.md 有具体任务** - Ralph 能立即开始任务 1 吗？
- [ ] **复杂功能有 specs/** - PROMPT.md 中有什么太模糊的吗？

如果您对这些都回答"是"，Ralph 会做得很好。

## 快速参考

| 需要... | 放在... |
|------------|--------------|
| 设置项目愿景和原则 | PROMPT.md |
| 定义技术栈 | PROMPT.md |
| 列出具体的实施任务 | fix_plan.md |
| 记录复杂功能需求 | specs/feature-name.md |
| 建立编码约定 | specs/stdlib/convention-name.md |
| 配置 Ralph 行为 | .ralphrc |
