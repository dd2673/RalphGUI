#### 📋 退出场景部分
使用 Given/When/Then 格式的六个具体场景：

**场景 1: 成功的项目完成**
- **给定**: 所有 fix_plan.md 项目标记为 [x]，测试通过，无错误
- **然后**: 输出 EXIT_SIGNAL=true，状态为 COMPLETE
- **Ralph 的操作**: 优雅退出循环并显示成功消息

**场景 2: 检测到仅测试循环**
- **给定**: 最近 3 次循环仅运行测试，无实施
- **然后**: 输出 WORK_TYPE=TESTING，FILES_MODIFIED=0
- **Ralph 的操作**: 递增 test_only_loops，达到阈值后退出

**场景 3: 卡在重复错误上**
- **给定**: 最近 5 次循环出现相同错误，无进度
- **然后**: 输出 STATUS=BLOCKED 及错误描述
- **Ralph 的操作**: 5 次循环后断路器打开

**场景 4: 没有剩余工作**
- **给定**: 所有任务完成，specs/ 中没有需要实施的内容
- **然后**: 输出 EXIT_SIGNAL=true，状态为 COMPLETE
- **Ralph 的操作**: 立即优雅退出

**场景 5: 正在取得进度**
- **给定**: 任务剩余，文件被修改，测试通过
- **然后**: 输出 STATUS=IN_PROGRESS 及进度指标
- **Ralph 的操作**: 继续循环，电路保持 CLOSED

**场景 6: 被外部依赖阻止**
- **给定**: 需要外部 API/库/人类决策
- **然后**: 输出 STATUS=BLOCKED 及具体阻止者
- **Ralph 的操作**: 记录阻止者，多次阻止后可能退出

**SMART 标准合规**:
- ✅ **Specific**: 每个场景都有精确条件
- ✅ **Measurable**: 布尔检查，可计数指标
- ✅ **Achievable**: 可进行自动检测
- ✅ **Relevant**: 直接解决退出检测问题
- ✅ **Timely**: 清楚何时应用条件

### 2. 用例文档 ✅
**专家建议**: Alistair Cockburn (用例方法论)
**创建的文件**: `USE_CASES.md` (600 行)

**内容**:

#### 参与者目录
- **Ralph** (主要参与者): 编排开发循环的 autonomous 代理
- **Claude Code** (支持参与者): AI 开发引擎
- **人类开发者** (支持参与者): 启动者和审查者

#### 六个主要用例

**UC-1: 执行开发循环** (主要工作流)
- **前置条件**: PROMPT.md 存在，fix_plan.md 有任务
- **成功**: 任务完成，文件被修改/提交，状态被跟踪
- **14 步主场景**，包含以下扩展:
  - 断路器 OPEN → 停止并提供指导
  - 速率限制超出 → 倒计时等待
  - API 5 小时限制 → 用户提示
  - 执行失败 → 带退避的重试
  - 检测到 EXIT_SIGNAL → 优雅完成
  - 断路器打开 → 停滞停止

**UC-2: 检测项目完成** (响应分析)
- **成功**: 准确确定完成，评分置信度
- **7 步主场景**，包含以下扩展:
  - 无结构化输出 → 自然语言解析
  - IN_PROGRESS 状态 → 工作类型分析
  - BLOCKED 状态 → 干预建议
  - 高置信度 → 即使没有显式信号也退出

**UC-3: 防止资源浪费** (断路器)
- **成功**: 停止失控循环，浪费 <1K tokens
- **9 步主场景**，包含以下扩展:
  - 文件未更改 (1 次循环) → 监控
  - 文件未更改 (2 次循环) → HALF_OPEN 警告
  - 文件未更改 (3 次循环) → OPEN 并停止
  - 相同错误 (5 次循环) → OPEN 并停止
  - 文件已更改 → 恢复到 CLOSED

**UC-4: 处理 API 速率限制**
- **成功**: 遵守速率限制，执行继续
- **9 步主场景**，包含以下扩展:
  - 新小时 → 重置计数器
  - 达到限制 → 倒计时等待
  - API 错误 → 带用户提示的重试

**UC-5: 提供循环监控** (ralph-monitor)
- **成功**: 实时可见状态，<2s 延迟
- **9 步持续监控**，包含以下扩展:
  - 无 status.json → 等待消息
  - 电路 OPEN → 红色警报显示
  - Ralph 退出 → 完成摘要

**UC-6: 重置断路器** (手动干预)
- **成功**: 电路重置，Ralph 可以恢复
- **11 步手动恢复**，包含以下扩展:
  - 无法确定原因 → 状态命令
  - PROMPT.md 问题 → 编辑和澄清
  - 环境问题 → 修复配置

#### 目标层次结构
```
系统目标: 以最小的 token 浪费完成项目
├─ 执行循环 (UC-1)
├─ 检测完成 (UC-2)
├─ 防止浪费 (UC-3)
├─ 遵守限制 (UC-4)
└─ 提供可见性 (UC-5)
```

#### 成功指标
| 用例 | 标准 | 目标 |
|----------|----------|--------|
| UC-1 | 完成率 | >95% |
| UC-2 | 检测准确度 | >90% |
| UC-3 | 电路跳闸时间 | <3 次循环 |
| UC-4 | 速率合规性 | 100% |
| UC-5 | 更新延迟 | <2s |
