# Ralph 用例

**作者**: 基于 Alistair Cockburn 的用例方法论
**日期**: 2025-10-01
**目的**: 为 Ralph autonomous 开发系统定义参与者、目标和场景

---

## 系统概述

**系统名称**: Ralph - Autonomous AI 开发循环
**系统目标**: 以最小的人工干预和 token 浪费完成软件项目实施
**主要参与者**: Ralph (编排 Claude Code 的 bash 脚本)
**支持参与者**: Claude Code (AI 开发引擎), 人类开发者 (启动者和审查者)

---

## 参与者目录

### 主要参与者: Ralph (Autonomous 代理)
**类型**: 系统
**目标**: 执行开发循环直到项目完成或断路器打开
**能力**:
- 使用 PROMPT.md 指令执行 Claude Code
- 分析 Claude Code 响应的完成信号
- 跟踪文件更改和进度
- 管理速率限制 (100 次调用/小时)
- 通过断路器检测停滞
- 工作完成时优雅退出

**约束**:
- 不能修改项目需求
- 必须遵守 API 速率限制
- 断路器打开时不能覆盖
- 需要有效的 PROMPT.md 和 fix_plan.md

---

### 支持参与者: Claude Code
**类型**: AI 系统
**目标**: 根据 PROMPT.md 指令实施功能、修复错误、运行测试
**能力**:
- 读取/写入/编辑文件
- 执行 bash 命令
- 运行测试并分析结果
- 搜索代码库
- 输出结构化状态报告

**约束**:
- 5 小时每日 API 限制
- Token 上下文限制
- 不能访问外部网络 (除经批准的工具外)
- 必须遵循 PROMPT.md 指令

---

### 支持参与者: 人类开发者
**类型**: 人类
**目标**: 启动 Ralph，在需要时审查结果和干预
**能力**:
- 创建 PROMPT.md 和 fix_plan.md
- 启动/停止 Ralph 执行
- 重置断路器
- 审查代码更改
- 在被阻止时提供澄清

**约束**:
- 在 autonomous 循环执行期间不在场
- Ralph 运行时不能修改文件
- 合并前必须审查更改

---

## 用例层次结构

### 系统目标: 完成项目实施
**子目标**:
1. 执行开发循环 (UC-1)
2. 检测完成条件 (UC-2)
3. 防止资源浪费 (UC-3)
4. 处理错误条件 (UC-4)
5. 提供可观测性 (UC-5)

---

## UC-1: 执行开发循环

**主要参与者**: Ralph
**利益相关者**: 人类开发者 (需要进度), Claude Code (执行者)
**前置条件**:
- PROMPT.md 存在且有效
- fix_plan.md 存在且至少有一个任务
- Claude Code CLI 已安装且可访问
- git 仓库已初始化

**成功保证** (后置条件):
- 一个开发任务完成
- 文件被修改并提交 (如果有更改)
- 状态在日志和 status.json 中跟踪
- 断路器状态更新
- 退出信号被分析并记录

**主要成功场景**:
1. Ralph 读取 PROMPT.md
2. Ralph 检查断路器状态 (必须为 CLOSED 或 HALF_OPEN)
3. Ralph 验证速率限制允许执行
4. Ralph 使用 PROMPT.md 执行 Claude Code
5. Claude Code 读取 fix_plan.md 并选择任务
6. Claude Code 实施任务 (文件被修改)
7. Claude Code 运行相关测试
8. Claude Code 输出 RALPH_STATUS 块
9. Ralph 分析 Claude 的响应 (analyze_response)
10. Ralph 更新 .exit_signals 文件 (update_exit_signals)
11. Ralph 在断路器中记录循环结果 (record_loop_result)
12. Ralph 递增调用计数器
13. Ralph 记录完成到 status.json 和 logs/
14. Ralph 继续下一个循环 (如果没有退出条件)

**扩展** (替代流):

**2a. 断路器为 OPEN**:
- 2a1. Ralph 显示断路器状态
- 2a2. Ralph 向用户显示指导 (检查日志、重置等)
- 2a3. Ralph 以退出代码 1 退出
- 用例结束

**3a. 速率限制超出**:
- 3a1. Ralph 计算到下一个小时重置的时间
- 3a2. Ralph 显示倒计时
- 3a3. Ralph 等待重置
- 3a4. Ralph 继续步骤 4

**3b. API 5 小时限制达到**:
- 3b1. Ralph 在 Claude 输出中检测到"速率限制"错误
- 3b2. Ralph 提示用户: 重试还是退出?
- 3b3a. 用户选择重试: 等待 5 分钟，转到步骤 4
- 3b3b. 用户选择退出: Ralph 优雅退出
- 用例结束

**4a. Claude Code 执行失败**:
- 4a1. Ralph 将错误记录到 logs/ralph_error.log
- 4a2. Ralph 用"失败"状态更新 status.json
- 4a3. Ralph 继续下一个循环 (重试)
- 4a4. 如果 5 次连续失败: 断路器打开
- 继续步骤 2

**9a. 响应分析检测到 EXIT_SIGNAL=true**:
- 9a1. Ralph 记录成功完成
- 9a2. Ralph 用"完成"状态更新 status.json
- 9a3. Ralph 显示完成摘要
- 9a4. Ralph 以退出代码 0 退出
- 用例结束

**11a. 断路器打开 (未检测到进度)**:
- 11a1. Ralph 记录断路器打开
- 11a2. Ralph 用"circuit_open"状态更新 status.json
- 11a3. Ralph 向用户显示指导
- 11a4. Ralph 以退出代码 1 退出
- 用例结束

**频率**: 持续循环直到完成或退出条件
**性能**: 在正常条件下每个循环应在 < 5 分钟内完成

## UC-2: 检测项目完成

**主要参与者**: Ralph (通过 response_analyzer.sh)
**利益相关者**: 人类开发者 (需要可靠退出), Claude Code (信号完成)
**前置条件**:
- 开发循环已执行 (UC-1)
- Claude Code 已产生输出

**成功保证**:
- 完成状态被准确确定
- .exit_signals 文件用决定更新
- 置信度分数计算 (0-100+)
- EXIT_SIGNAL 正确设置 (true/false)

**主要成功场景**:
1. Ralph 读取 Claude Code 输出文件
2. Ralph 检查结构化 RALPH_STATUS 块
3. Ralph 发现 STATUS: COMPLETE 和 EXIT_SIGNAL: true
4. Ralph 将置信度分数设置为 100
5. Ralph 在 .response_analysis 中将 exit_signal 设置为 true
6. Ralph 用 done_signals 数组更新 .exit_signals
7. Ralph 在下一次循环检查中触发优雅退出

**扩展**:

**2a. 未找到结构化输出**:
- 2a1. Ralph 搜索自然语言完成关键字
- 2a2. 如果找到: 置信度分数 +10
- 2a3. Ralph 检查"无事可做"模式
- 2a4. 如果找到: 置信度分数 +15，设置 exit_signal=true
- 继续步骤 6

**3a. STATUS 显示 IN_PROGRESS**:
- 3a1. Ralph 检查 WORK_TYPE 字段
- 3a2. 如果连续第 3 次 WORK_TYPE=TESTING: 标记为 test_only
- 3a3. 如果连续第 3 次 FILES_MODIFIED=0: 断路器打开
- 3a4. 将 exit_signal 设置为 false
- 继续步骤 6

**3b. STATUS 显示 BLOCKED**:
- 3b1. Ralph 递增 blocked_loops 计数器
- 3b2. 如果 blocked_loops >= 3: 推荐人工干预
- 3b3. 将 exit_signal 设置为 false
- 继续步骤 6

**6a. 置信度分数 >= 40**:
- 6a1. 即使没有显式 EXIT_SIGNAL，也设置 exit_signal=true
- 6a2. 记录高置信度完成检测
- 继续步骤 7

**频率**: 每个开发循环后
**性能**: 分析应在 < 1 秒内完成

## UC-3: 防止资源浪费 (断路器)

**主要参与者**: Ralph (通过 circuit_breaker.sh)
**利益相关者**: 人类开发者 (想要避免 token 浪费)
**前置条件**:
- 开发循环正在执行
- 断路器已初始化

**成功保证**:
- 检测并停止失控循环
- 最小化 token 浪费 (< 1K 浪费的 tokens)
- 提供清晰的停止用户指导
- 断路器状态在重启间持久化

**主要成功场景**:
1. Ralph 将断路器初始化为 CLOSED 状态
2. 每次循环后，Ralph 调用 record_loop_result()
3. Ralph 从 git diff 计算 files_changed
4. Ralph 从 Claude 输出检测 has_errors
5. Ralph 计算 output_length
6. 断路器更新 consecutive_no_progress 计数器
7. consecutive_no_progress 为 0 (检测到进度)
8. 断路器保持 CLOSED
9. Ralph 继续下一个循环

**扩展**:

**6a. 无文件更改 (consecutive_no_progress 递增)**:
- 6a1. consecutive_no_progress = 1
- 6a2. 断路器保持 CLOSED
- 继续步骤 9

**6b. 第 2 次连续无文件更改**:
- 6b1. consecutive_no_progress = 2
- 6b2. 断路器转换到 HALF_OPEN
- 6b3. Ralph 记录"监控模式"警告
- 继续步骤 9

**6c. 第 3 次连续无文件更改**:
- 6c1. consecutive_no_progress = 3
- 6c2. 断路器转换到 OPEN
- 6c3. Ralph 显示带有指导的停止消息
- 6c4. Ralph 以退出代码 1 退出
- 用例结束

**6d. 第 5 次连续检测到相同错误**:
- 6d1. consecutive_same_error = 5
- 6d2. 断路器转换到 OPEN
- 6d3. 原因: "相同错误在 5 次连续循环中重复"
- 继续步骤 6c3

**7a. 检测到文件更改 (恢复)**:
- 7a1. consecutive_no_progress 重置为 0
- 7a2. 如果电路为 HALF_OPEN: 转换到 CLOSED
- 7a3. Ralph 记录"电路恢复"
- 继续步骤 9

**频率**: 每次开发循环后
**性能**: 断路器检查 < 100ms

## UC-4: 处理 API 速率限制

**主要参与者**: Ralph
**利益相关者**: 人类开发者 (需要不间断执行)
**前置条件**:
- Ralph 正在执行开发循环
- 调用跟踪已初始化

**成功保证**:
- API 速率限制被遵守
- 调用计数器准确跟踪
- 每小时重置自动处理
- 用户被告知等待时间

**主要成功场景**:
1. Ralph 检查当前小时 (YYYYMMDDHH 格式)
2. Ralph 读取 .last_reset 时间戳
3. 当前小时匹配 last_reset (同一小时)
4. Ralph 读取 .call_count
5. call_count 为 45 (< 100 限制)
6. Ralph 允许执行
7. Ralph 将 call_count 递增到 46
8. Ralph 将更新后的计数写入 .call_count
9. 执行继续

**扩展**:

**3a. 检测到新小时 (小时改变)**:
- 3a1. Ralph 将 call_count 重置为 0
- 3a2. Ralph 将当前小时写入 .last_reset
- 3a3. Ralph 记录"新小时的调用计数器重置"
- 继续步骤 5

**5a. call_count 等于或超过限制 (100)**:
- 5a1. Ralph 计算到下一小时的秒数
- 5a2. Ralph 显示倒计时: "速率限制达到。等待 HH:MM:SS..."
- 5a3. Ralph 休眠计算的时间
- 5a4. Ralph 重置计数器 (转到步骤 3a1)
- 继续步骤 6

**5b. Claude 返回 API 速率限制错误**:
- 5b1. Ralph 在输出中检测到"rate_limit_error"
- 5b2. Ralph 提示: "达到 API 5 小时限制。重试？ (y/n)"
- 5b3a. 用户输入 'y': Ralph 等待 5 分钟，重试
- 5b3b. 用户输入 'n': Ralph 优雅退出
- 用例结束

**频率**: 每次 Claude Code 执行前
**性能**: 速率限制检查 < 50ms

## UC-5: 提供循环监控

**主要参与者**: ralph-monitor.sh
**利益相关者**: 人类开发者 (需要实时可见性)
**前置条件**:
- Ralph 正在运行 (ralph_loop.sh)
- ralph-monitor 在单独终端启动

**成功保证**:
- 实时状态显示和更新
- 循环计数、速率限制和进度可见
- 断路器状态显示
- 退出信号被跟踪

**主要成功场景**:
1. 用户在单独终端启动 ralph-monitor.sh
2. 监控每 2 秒读取 status.json
3. 监控显示循环计数、状态、时间戳
4. 监控读取 .call_count 并显示 "Calls: 45/100"
5. 监控读取 .circuit_breaker_state 并显示状态
6. 监控读取 .exit_signals 并显示信号计数
7. 监控检测 status.json 更新
8. 监控用新数据刷新显示
9. 循环继续 (转到步骤 2)

**扩展**:

**3a. status.json 尚不存在**:
- 3a1. 监控显示"等待 Ralph 启动..."
- 3a2. 监控休眠 2 秒
- 继续步骤 2

**5a. 断路器为 OPEN**:
- 5a1. 监控以红色显示状态
- 5a2. 监控显示电路打开的原因
- 5a3. 监控显示"执行停止"消息
- 继续步骤 7

**7a. Ralph 已退出**:
- 7a1. 监控检测最终状态
- 7a2. 监控显示完成摘要
- 7a3. 监控显示总循环数、持续时间、退出原因
- 7a4. 监控退出
- 用例结束

**频率**: 持续直到 Ralph 退出
**性能**: 更新延迟 < 2 秒

## UC-6: 重置断路器 (手动干预)

**主要参与者**: 人类开发者
**利益相关者**: Ralph (需要手动重置才能继续)
**前置条件**:
- 断路器为 OPEN
- Ralph 已停止执行
- 用户已审查日志并识别问题

**成功保证**:
- 断路器重置为 CLOSED 状态
- 计数器重置为 0
- Ralph 可以恢复执行
- 重置原因被记录

**主要成功场景**:
1. 用户识别断路器已打开 (从 ralph-monitor 或日志)
2. 用户审查 logs/ralph.log 了解原因
3. 用户修复底层问题 (更新 fix_plan.md、修复错误等)
4. 用户运行: `ralph --reset-circuit`
5. Ralph 加载 circuit_breaker.sh 函数
6. Ralph 调用 reset_circuit_breaker("用户手动重置")
7. Ralph 在 .circuit_breaker_state 中将状态设置为 CLOSED
8. Ralph 将所有计数器重置为 0
9. Ralph 记录"断路器重置为 CLOSED 状态"
10. Ralph 显示成功消息
11. 用户现在可以重启 Ralph 执行

**扩展**:

**2a. 用户无法从日志确定原因**:
- 2a1. 用户运行: `ralph --status` 获取更多信息
- 2a2. 用户检查 .circuit_breaker_history 的状态转换
- 2a3. 用户审查最近的 Claude 输出文件
- 继续步骤 3

**3a. 问题在 PROMPT.md 或 specs/**:
- 3a1. 用户编辑 PROMPT.md 澄清需求
- 3a2. 用户在 specs/ 中更新缺失信息
- 3a3. 用户提交更改
- 继续步骤 4

**3b. 问题是配置或环境**:
- 3b1. 用户安装缺失的依赖
- 3b2. 用户修复环境变量
- 3b3. 用户验证配置
- 继续步骤 4

**频率**: 需要时 (当断路器打开时)
**性能**: 重置是瞬时的

## 目标层次结构

```
系统目标: 以最小的 token 浪费完成项目实施
├─ 子目标 1: 执行开发循环 (UC-1)
│  ├─ 成功: 文件更改，测试通过，任务完成
│  └─ 失败: 文件未更改，测试失败，无进度
│
├─ 子目标 2: 检测何时无法再取得进度 (UC-2)
│  ├─ 成功: 优雅退出并显示完成摘要
│  └─ 失败: 工作完成时继续循环
│
├─ 子目标 3: 防止资源浪费 (UC-3)
│  ├─ 成功: 停滞时停止执行
│  └─ 失败: 在无限循环中燃烧 tokens
│
├─ 子目标 4: 遵守 API 限制 (UC-4)
│  ├─ 成功: 等待重置，无缝继续
│  └─ 失败: 超出限制，API 错误
│
└─ 子目标 5: 提供可见性 (UC-5)
   ├─ 成功: 用户有实时状态
   └─ 失败: 黑盒，无反馈
```

## 非功能性需求

### 可靠性
- **可用性**: 网络和 API 可用时 99%+
- **容错性**: 优雅处理 Claude API 错误
- **数据完整性**: 意外终止时无数据丢失

### 性能
- **响应时间**: 状态检查 < 100ms
- **吞吐量**: 支持连续运行数天
- **可扩展性**: 处理 100+ 次循环的项目

### 可用性
- **可学习性**: 新用户在 < 30 分钟内理解系统
- **错误消息**: 失败时清晰、可操作的指导
- **文档**: 完整的用例和示例

### 安全性
- **身份验证**: 遵守 Claude API 身份验证
- **授权**: 仅对授权文件操作
- **数据隐私**: 不记录敏感数据

## 术语表

| 术语 | 定义 |
|------|------------|
| **断路器** | 通过检测停滞防止失控循环的模式 |
| **退出信号** | Claude 已完成所有工作的指示器 |
| **循环** | Ralph 执行 Claude Code 的一次迭代 |
| **速率限制** | 每小时允许的最大 API 调用次数 (100) |
| **响应分析器** | 解析 Claude 输出以获取信号的组件 |
| **停滞** | 未取得进度的条件 (无文件更改) |
| **仅测试循环** | 仅运行测试、无实施工作的循环 |
