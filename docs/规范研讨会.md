# Ralph 规范研讨会指南

**基于**: Janet Gregory 的"三好友"协作测试方法
**目的**: 为新 Ralph 功能促进富有成效的规范讨论
**受众**: 从事 Ralph 增强的开发人员、测试人员、产品负责人

---

## 什么是规范研讨会？

规范研讨会汇集了三个视角 ("三好友") 来在实施前定义功能：

1. **开发人员** (如何实现) - 技术可行性和方法
2. **测试人员** (如何验证) - 边界情况、验证、质量标准
3. **产品负责人/用户** (价值是什么) - 业务需求和成功标准

**目标**: 生成具体的、可测试的规范，防止错误和误解。

## 示例研讨会: 速率限制自动重试

**功能**: API 速率限制错误时自动重试

### 1. 用户故事

**作为** Ralph 用户
**我想要** 在临时 API 错误时自动重试
**以便** 瞬态问题不会停止我的开发工作流

### 2. 验收标准

- [x] Ralph 在 Claude 输出中检测到 "rate_limit_error"
- [x] Ralph 在重试前等待适当时间 (5 分钟)
- [x] Ralph 将重试限制为 3 次
- [x] Ralph 在持续失败时回退到用户提示
- [x] 重试尝试被清晰地记录

### 3. 测试人员的问题

**问**: 什么算作"速率限制错误" vs 其他错误？
**答**: 输出中的特定字符串 "rate_limit_error" 或 "429" 状态码

**问**: 重试是否计入每小时调用限制？
**答**: 是的，重试尝试消耗调用配额

**问**: 如果用户在等待期间按 Ctrl+C 会怎样？
**答**: 优雅关闭，保存状态，允许恢复

### 4. 实施方法

**方法**:
- 在 `execute_claude_code()` 函数中添加重试逻辑
- 实现指数退避 (5 分钟 → 10 分钟 → 15 分钟)
- 在 `.retry_state` 文件中存储重试状态
- 在 status.json 中添加重试计数器

**约束**:
- 必须与现有速率限制跟踪配合使用
- 不能绕过断路器
- 重试必须遵守 API 5 小时限制

### 5. 示例化规范

**场景 1: 成功重试**

**给定**:
- Ralph 在循环 #5 执行 Claude Code
- Claude 返回 "rate_limit_error: please retry"
- 重试计数为 0

**当**: Ralph 检测到速率限制错误

**那么**:
- Ralph 记录 "检测到速率限制，第 1/3 次尝试。等待 5 分钟..."
- Ralph 休眠 300 秒
- Ralph 重试 Claude Code 执行
- 如果成功: 正常继续，将重试计数重置为 0

**场景 2: 持续失败**

**给定**:
- Ralph 已经重试了 3 次
- 每次重试都导致 "rate_limit_error"

**当**: 第 4 次执行也返回速率限制错误

**那么**:
- Ralph 记录 "超过重试限制 (3 次尝试)"
- Ralph 提示用户: "继续等待？ (y/n)"
- 用户决定决定下一步操作 (退出或继续)

### 6. 边界情况

1. 第一次循环时速率限制错误 → 立即重试
2. 用户在等待期间中断 → 干净关闭，状态保留
3. 重试后出现不同错误 → 作为正常错误处理，不增加重试计数
4. 速率限制在第 1 次重试后解除 → 重置计数器，正常继续

### 7. 测试策略

**单元测试**:
- [x] 测试重试检测逻辑
- [x] 测试指数退避计算
- [x] 测试重试限制执行

**集成测试**:
- [x] 模拟速率限制错误，验证重试发生
- [x] 模拟 3 次失败，验证回退到用户提示
- [x] 验证重试状态在重启间持久化

### 8. 完成定义

- [x] 在 ralph_loop.sh 中实施代码
- [x] 在 tests/unit/ 中添加单元测试
- [x] 在 tests/integration/ 中添加集成测试
- [x] 在 README.md 中更新文档

## 研讨会最佳实践

### 研讨会前
1. **准备**: 提前 24 小时向参与者发送用户故事
2. **背景**: 提供相关背景 (为什么现在需要此功能？)
3. **时间盒**: 安排最多 30-60 分钟

### 研讨会期间
1. **专注**: 一次处理一个功能
2. **具体**: 使用真实示例，不是抽象描述
3. **问题**: 鼓励测试人员问"什么可能出错？"
4. **记录**: 实时捕获决策

### 研讨会后
1. **总结**: 向所有参与者发送笔记
2. **跟踪**: 为行动项目创建任务
3. **参考**: 使用场景作为测试用例

### 危险信号
❌ "我们会在实施过程中想办法"
❌ "那是边界情况，我们稍后处理"
❌ 模糊的验收标准
❌ 没有具体示例
❌ 跳过测试人员视角

### 成功指标
✅ 清晰的、可测试的场景
✅ 在编码前识别边界情况
✅ 代表所有三个视角
✅ 具体示例，不是抽象
✅ 参与者之间有共同理解
