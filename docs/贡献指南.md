## 入门

### 前置要求

在贡献之前，确保您已安装以下软件：

- **Python 3.10+** - 用于脚本执行
- **Git** - 版本控制 (必需)
- **PySide6** - 用于 GUI 开发
- **PyYAML** - 用于配置

### 克隆仓库

```bash
# 首先在 GitHub 上 fork 仓库，然后克隆您的 fork
git clone https://github.com/YOUR_USERNAME/ralph-gui.git
cd ralph-gui
```

### 安装依赖

```bash
# 安装 Python 依赖
pip install -r requirements.txt

# 验证 Python 版本
python --version  # 应该显示 Python 3.10+

# 验证依赖
python -c "import PySide6; print('PySide6 installed')"
```

### 验证您的设置

```bash
# 运行测试套件以确保一切正常
pytest

# 运行应用程序
python src/ralph_gui/main.py
```

### 项目结构

```
ralph-gui/
├── src/ralph_gui/         # 主应用程序
│   ├── main.py           # 应用程序入口点
│   ├── app.py            # RalphApp 应用程序类
│   ├── main_window.py    # 主窗口 (MVP View)
│   ├── presenters/       # Presenter 层
│   ├── services/         # Service 层
│   ├── models/           # Model 层
│   ├── views/            # View 层
│   ├── lib/              # 库模块
│   ├── scripts/          # Python 脚本
│   └── templates/        # 模板文件
├── scripts/              # 主脚本目录
├── lib/                  # 核心库模块
├── tests/                # 测试套件
├── docs/                 # 文档
└── wrappers/             # Windows 批处理脚本包装器
```

## 开发工作流程

### 分支命名约定

始终创建功能分支 - 切勿直接在 `main` 上工作：

| 分支类型 | 格式 | 示例 |
|-------------|--------|---------|
| 新功能 | `feature/<功能名称>` | `feature/log-rotation` |
| 错误修复 | `fix/<问题名称>` | `fix/rate-limit-reset` |
| 文档 | `docs/<文档更新>` | `docs/api-reference` |
| 测试 | `test/<测试区域>` | `test/circuit-breaker` |
| 重构 | `refactor/<区域>` | `refactor/response-analyzer` |

```bash
# 创建新的功能分支
git checkout -b feature/my-awesome-feature
```

### 提交消息格式

我们使用 [Conventional Commits](https://www.conventionalcommits.org/) 来保持清晰、结构化的提交历史：

```
<类型>(<范围>): <描述>

[可选正文]

[可选页脚]
```

**类型：**

| 类型 | 描述 | 示例 |
|------|-------------|---------|
| `feat` | 新功能 | `feat(loop): add dry-run mode` |
| `fix` | 错误修复 | `fix(monitor): correct refresh rate` |
| `docs` | 仅文档 | `docs(readme): update installation steps` |
| `test` | 添加/更新测试 | `test(setup): add template validation tests` |
| `refactor` | 代码更改 (无功能/修复) | `refactor(analyzer): simplify error detection` |
| `chore` | 维护任务 | `chore(deps): update PySide6 version` |

**示例：**

```bash
# 功能添加
feat(loop): add dry-run mode for testing

# 带范围的错误修复
fix(gui): correct circuit breaker status display

# 文档更新
docs(status): update IMPLEMENTATION_STATUS.md

# 测试添加
test(cli): add comprehensive CLI parsing tests
```

**编写好的提交消息：**

- 使用祈使语气 ("add" 而不是 "added")
- 解释 WHAT 更改和 WHY (不是 HOW)
- 主题行保持在 72 个字符以内
- 适当时引用 issues (`fixes #123`)

## 代码风格指南

### Python 最佳实践

Ralph GUI 遵循一致的 Python 约定：

**文件结构：**

```python
"""模块描述。"""

import standard_library
import third_party

# 常量 (UPPER_CASE)
MAX_CALLS_PER_HOUR = 100
CB_NO_PROGRESS_THRESHOLD = 3

class MyClass:
    """类描述。"""

    def __init__(self, param1: str, param2: int):
        self.param1 = param1
        self.param2 = param2

def helper_function(param1: str) -> bool:
    """函数描述。

    Args:
        param1: param1 的描述。

    Returns:
        返回值的描述。
    """
    return True
```

**命名约定：**

| 元素 | 约定 | 示例 |
|---------|------------|---------|
| 函数 | snake_case | `get_circuit_state()` |
| 类 | PascalCase | `CircuitBreakerModel` |
| 常量 | UPPER_SNAKE_CASE | `MAX_CALLS_PER_HOUR` |
| 变量 | snake_case | `loop_count = 0` |
| 私有方法 | _snake_case | `_private_method()` |
| 文件名 | snake_case.py | `circuit_breaker.py` |
| 控制文件 | snake_case.md | `fix_plan.md`, `AGENT.md` |

**Docstring 风格：**

```python
def my_function(param1: str, param2: int) -> bool:
    """函数的简短描述。

    更长的描述（如果需要），解释函数的用途
    和行为。

    Args:
        param1: 第一个参数的描述。
        param2: 第二个参数的描述。

    Returns:
        返回值的描述。

    Raises:
        ValueError: 引发此异常的描述。

    Example:
        >>> my_function("test", 42)
        True
    """
    pass
```

**错误处理：**

```python
# 始终验证输入
if not param1:
    raise ValueError("param1 is required")

# 使用特定异常
if not file_exists(path):
    raise FileNotFoundError(f"File not found: {path}")

# 优雅地处理异常
try:
    result = risky_operation()
except SpecificError as e:
    logger.error(f"Specific error occurred: {e}")
    raise
except Exception as e:
    logger.exception("Unexpected error")
    raise
```

## 测试要求

### 强制性测试标准

**所有新功能必须包含测试。这是不可商量的。**

| 要求 | 标准 | 执行 |
|-------------|----------|-------------|
| 测试通过率 | 100% | **强制** - CI 阻止合并 |
| 测试覆盖率 | 80% | 鼓舞人心 - 仅供参考 |

### 测试组织

```
tests/
├── unit/                       # 快速、隔离的测试
│   ├── test_circuit_breaker.py
│   ├── test_response_analyzer.py
│   └── test_rate_limiting.py
├── integration/                # 多组件测试
│   ├── test_loop_execution.py
│   └── test_cli_service.py
└── conftest.py                # 共享 fixtures
```

### 运行测试

| 命令 | 目的 | 使用时机 |
|---------|---------|---------|
| `pytest` | 运行所有测试 | 提交前、PR 前 |
| `pytest tests/unit/` | 仅单元测试 | 开发期间 |
| `pytest tests/integration/` | 仅集成测试 | 测试交互 |
| `pytest tests/test_file.py::test_name` | 单个测试 | 调试特定测试 |

### 编写测试

**测试结构：**

```python
"""我的功能的单元测试。"""

import pytest
from ralph_gui.models.circuit_breaker import CircuitBreakerModel

class TestCircuitBreakerModel:
    """CircuitBreakerModel 的测试。"""

    def setup_method(self):
        """每个测试前设置。"""
        self.cb = CircuitBreakerModel()

    def teardown_method(self):
        """每个测试后清理。"""
        pass

    def test_initial_state_is_closed(self):
        """测试初始断路器状态为 CLOSED。"""
        assert self.cb.state == "CLOSED"

    def test_can_make_call_under_limit(self):
        """测试在限制下 can_make_call 返回 True。"""
        assert self.cb.can_make_call(50, 100) is True

    def test_can_make_call_at_limit(self):
        """测试在限制下 can_make_call 返回 False。"""
        assert self.cb.can_make_call(100, 100) is False
```

**测试最佳实践：**

1. **测试成功和失败情况**
2. **使用描述性测试名称** 说明场景
3. **隔离测试** - 每个测试应该是独立的
4. **模拟外部依赖** (文件系统、CLI 命令等)
5. **测试边界情况** (空文件、无效输入、边界值)
6. **为复杂测试场景** 添加 docstrings

## Pull Request 流程

### 创建 PR 前

运行此检查清单：

- [ ] 所有测试在本地通过 (`pytest`)
- [ ] 新代码包含适当的测试
- [ ] 提交遵循 conventional 格式
- [ ] 需要时更新文档
- [ ] 无调试代码或 print 语句
- [ ] 无提交 secrets 或凭据

### 创建 PR

1. **推送您的分支：**
   ```bash
   git push origin feature/my-feature
   ```

2. **在 GitHub 上打开 Pull Request，包含：**

**PR 标题：** 遵循 conventional 提交格式
```
feat(loop): add dry-run mode for testing
```

**PR 描述模板：**
```markdown
## 摘要

简要描述此 PR 的内容 (1-3 个要点)。

- 添加试运行模式以预览循环执行
- 包含新 CLI 标志 `--dry-run`
- 记录操作但不进行实际更改

## 测试计划

- [ ] 添加/更新单元测试
- [ ] 添加/更新集成测试
- [ ] 完成手动测试

## 相关 Issues

Fixes #123
Related to #456
```

### PR 创建后

1. **等待 CI/CD** - GitHub Actions 将运行所有测试
2. **处理审查反馈** - 及时进行要求的更改
3. **保持 PR 更新** - 如果主分支已更改则 rebase

## 代码审查指南

### 对于贡献者

**响应反馈：**

- 感谢审查人员的时间
- 如果要求不清楚则提问
- 及时进行要求的更改
- 随着更改的发展更新 PR 描述
- 不要个人化反馈 - 这是关于代码的

**如果您不同意：**

- 清晰地解释您的推理
- 为您的决定提供背景
- 对替代方法持开放态度
- 有疑问时服从维护者的判断

### 对于审查者

**检查内容：**

| 区域 | 要问的问题 |
|------|------------------|
| **正确性** | 代码是否做了它声称的事情？ |
| **测试** | 测试是否全面？是否通过？ |
| **风格** | 是否遵循 Python 约定？ |
| **文档** | docstrings 和文档是否更新？ |
| **破坏性更改** | 这会影响现有用户吗？ |
| **性能** | 有明显的性能问题吗？ |

**审查最佳实践：**

1. **建设性** - 专注于改进，而不是批评
2. **具体** - 尽可能指向确切行
3. **解释原因** - 帮助贡献者学习
4. **认可好的工作** - 注意编写良好的代码
5. **准备好时批准** - 不要扣留 PR

## 质量标准

### 质量门禁

所有 PR 必须通过这些自动化检查：

| 门禁 | 要求 | 执行 |
|------|-------------|-------------|
| 单元测试 | 100% 通过 | **阻止合并** |
| 集成测试 | 100% 通过 | **阻止合并** |
| 覆盖率 | 80% | 仅供参考 |
| Conventional Commits | 必需 | 手动审查 |
| 文档 | 已更新 | 手动审查 |

### 文档标准

**何时更新文档：**

- 添加新 CLI 标志 → 更新 README.md、CLAUDE.md
- 添加新功能 → 更新 README.md "功能"部分
- 更改行为 → 更新相关文档
- 添加新模式 → 更新 CLAUDE.md

**保持同步：**

1. **CLAUDE.md** - 技术规范、质量标准
2. **README.md** - 用户面向的文档、安装
3. **模板** - 保持模板文件最新
4. **Docstrings** - 更改代码时更新
